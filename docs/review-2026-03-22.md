# Review Memo (2026-03-22)

この文書は、`omujina-ros-CLI` の一時レビュー記録です。
まずローカル実装のコードレビュー結果を残し、その後 upstream / 関連リポジトリ調査と修正計画を追記します。

## 現時点の主要指摘

### P1

1. ジョブ実行が排他的でも冪等でもない
   - 対象: `src/mujina_assist/app.py`, `src/mujina_assist/services/jobs.py`
   - `worker --job-file` の二重起動や、同種ジョブの並走を防ぐ仕組みがなく、`setup` / `policy_switch` / `zero` のような破壊的処理が重複実行され得る。

2. ジョブグループ起動の途中失敗で半端状態が残る
   - 対象: `src/mujina_assist/app.py`
   - `SIM` や `実機起動` の複数ジョブ起動中に一部だけ失敗しても、先に起動したジョブを止めずに失敗を返すため、裏で一部プロセスだけ生き残る。

3. policy ロールバック失敗時に表示状態と実体がズレる
   - 対象: `src/mujina_assist/services/policy.py`
   - policy 切替失敗後の復旧ビルドが失敗しても、runtime state は元に戻したように見せるため、workspace 実体と UI 表示が乖離する。

### P2

4. 永続状態ファイルの障害時契約が不統一
   - 対象: `src/mujina_assist/services/state.py`, `src/mujina_assist/services/jobs.py`, `src/mujina_assist/services/policy.py`, `src/mujina_assist/services/checks.py`
   - `runtime.json` は破損で起動不能になる一方、job / policy index は黙って空扱いされる。壊れ方が一貫せず、観測と復旧が難しい。

5. 診断フローが前提失敗を止め切れていない
   - 対象: `src/mujina_assist/app.py`
   - `handle_robot_diagnostics()` が `handle_preflight()` の失敗を見ずに追加診断へ進めるため、原因の切り分けを濁す。

6. ターミナル起動失敗の観測性が弱い
   - 対象: `src/mujina_assist/services/terminals.py`
   - GUI terminal / `tmux` の起動例外や stderr を潰しており、「見つからない」と「起動に失敗した」が区別できない。

7. interactive 実行のログパス前提が脆い
   - 対象: `src/mujina_assist/services/shell.py`
   - `run_bash(..., interactive=True, log_path=...)` が `tee` の前にログディレクトリ作成を保証していない。

### P3

8. 「最近のジョブ」表示が時系列順ではない
   - 対象: `src/mujina_assist/services/jobs.py`
   - ファイル名を文字列逆順で並べているため、`kind` に引っ張られて最近順が崩れる。

9. IMU ポート推定が楽観的すぎる
   - 対象: `src/mujina_assist/services/checks.py`
   - `/dev/rt_usb_imu` が無いとき、単独の `ttyUSB*` / `ttyACM*` を IMU とみなしてしまう。

10. `can_mode` の不正値を fail-fast しない
    - 対象: `src/mujina_assist/services/processes.py`
    - typo や破損 job file でも黙って network CAN 側に倒れるため、設定不整合が見えづらい。

11. `start.sh` と README / テストの復旧系カバレッジが薄い
    - 対象: `start.sh`, `README.md`, `tests/test_start_sh.py`, `scripts/run-container-tests.sh`
    - エントリポイントの分岐や worker exit code 契約の検証が弱く、再実行や失敗時の保証が曖昧。

### upstream 側の重要前提

1. `mujina_ros` の基本要件は Ubuntu 24.04 + ROS 2 Jazzy + `rosdep` + `mujoco/torch/onnxruntime` の pip 導入。
2. 実機起動手順は 3 ターミナル前提:
   - IMU driver
   - `can_setup_*` + `mujina_main`
   - `joy_linux_node`
3. 初回利用前に motor origin 設定が必要。
4. `sim` は `mujina_main --sim` で行う。
5. policy / 強化学習は `mujina_isaac_lab` という別リポジトリ側に分かれている。
6. `90-mujina.rules` は upstream README に「device に合わせて編集」と明記されている。
7. `can_setup_serial.sh` は `slcand` と `/dev/usb_can` を前提にしている。

### ハードウェア / ドメイン側の重要前提

1. `Mujina_Hardware` 側では、IMU と USB-CAN に firmware 書き込みが必要。
2. モーター ID の書き込みと firmware 更新も別手順として存在する。
3. Mujina は LattePanda IOTA を内蔵し、auto power on 設定も前提に入る。
4. ハードウェア README には、未接続 XT30 コネクタの絶縁や配線手順など、運用安全に直結する注意がある。
5. IMU driver は「強化学習デプロイ用の簡易版」で、2000000bps の ASCII フォーマットを想定している。
6. 公式情報では Mujina は点検・巡回、遠隔操作、自動巡回代替などの用途を想定している。
7. 公開されているリスクアセスメントには、50cm 以上離れる、独立したモータ停止・電源停止、通信断時の安全遷移などの考え方が含まれる。

## 外部知識を踏まえた追加レビュー

### P1

12. 実機 safety gate がドメイン前提に対して弱い
    - 対象: `src/mujina_assist/app.py`
    - 現状は `REAL` / `ZERO ...` の文字入力確認と SIM 確認フラグが主な歯止めだが、公開されているリスクアセスメントの水準に比べると、離隔、補助者、非常停止、通信断・転倒時対応の確認が不足している。

### P2

13. udev / firmware / VID:PID 前提を setup に組み込めていない
    - 対象: `src/mujina_assist/services/workspace.py`, `src/mujina_assist/services/checks.py`
    - upstream README は `90-mujina.rules` の編集を要求し、Hardware README は IMU / USB-CAN firmware 書き込みを要求しているが、CLI はルールをそのままコピーして存在確認するだけで、実機固有差分の検証がない。

14. IMU readiness 判定が protocol まで見ていない
    - 対象: `src/mujina_assist/services/checks.py`, `src/mujina_assist/services/processes.py`
    - `rt_usb_imu_driver` は 2000000bps ASCII を想定する簡易 driver だが、CLI は `/dev/rt_usb_imu` があるかどうかしか見ない。誤 firmware や別 device でも「準備完了」に見せやすい。

15. policy 管理が「読み込める」以上の保証を持っていない
    - 対象: `src/mujina_assist/services/policy.py`, `src/mujina_assist/app.py`
    - `mujina_isaac_lab` が別 repo に分かれている以上、本来は学習元 task、ロボット revision、observation / action 契約、生成日時などの provenance が重要になるが、CLI は実質 ONNX load test と hash 管理止まり。

16. motor ID / 初回ゼロ点 / gamepad mode の現場前提が CLI に載っていない
    - 対象: `src/mujina_assist/app.py`, `src/mujina_assist/services/checks.py`
    - hardware 側では motor ID 書き込み手順と配線が重要で、upstream README では gamepad の X mode / LED off が前提だが、CLI 側の診断や setup 導線には組み込まれていない。

17. 自動化対象の境界が曖昧
    - 対象: `README.md`, `docs/*`, `src/mujina_assist/*`
    - Mujina の実運用前提は複数 repo・firmware・組立手順にまたがるのに、CLI がカバーする範囲と人手でやるべき範囲が明文化されていない。

## 全体修正計画

### フェーズ 0: 安全性と整合性の穴を先に塞ぐ

1. Job 実行を排他的にする
   - `job_file` 単位の lock file または OS lock を導入し、`queued -> running` を原子的にする。
2. ジョブグループ起動を transaction っぽく扱う
   - 途中失敗時は先行起動ジョブを停止するか、「partial_started」として明示記録する。
3. policy ロールバックの truthfulness を直す
   - 復旧 build に失敗したら `state` を「不整合」「手動復旧待ち」として残す。
4. 危険操作の checklist を導入する
   - `REAL` / `ZERO` 実行前に、補助者、周辺離隔、停止手段、所定姿勢、通信状態を明示チェックする。

### フェーズ 1: 実機セットアップを domain-aware にする

5. udev ルール適用を wizard 化する
   - 現物 VID:PID を検出し、`90-mujina.rules` の差分確認、適用、再読込、symlink 検証まで行う。
6. firmware / motor ID / gamepad mode の診断を追加する
   - IMU firmware, USB-CAN firmware, motor ID 配列, gamepad mode を doctor に取り込む。
7. 初回ゼロ点完了状態を明示管理する
   - ロボットごとに「zero 済み」状態を記録し、未完了なら実機起動前に強く止める。

### フェーズ 2: policy 運用を単なるファイル差し替えから昇格させる

8. policy manifest を導入する
   - 学習元 repo / task / 入出力次元 / hash / 作成日時 / 対応 robot revision を保持する。
9. 実機投入前の approval 状態を分ける
   - `imported`, `loadable`, `sim_verified`, `real_approved` のように段階を分ける。
10. `mujina_isaac_lab` との接続点を文書化する
    - export 手順、必要 manifest、CLI が期待する契約を明文化する。

### フェーズ 3: 観測性と復旧性を底上げする

11. JSON 永続化を atomic write + backup に変更する
    - `.tmp` へ書いて rename、破損時は `.bak` から復旧できるようにする。
12. terminal / shell error を潰さず残す
    - backend 名、stderr、return code を job metadata に記録する。
13. recent jobs と log 表示を直す
    - `created_at` 基準で並べ、巨大ログは全文読まず tail 取得する。
14. 失敗を silent empty にしない
    - policy index / jobs / runtime state の破損は warning と recovery hint を出す。

### フェーズ 4: テストとドキュメントを現実に寄せる

15. failure path テストを増やす
    - 二重 worker 実行、partial launch、corrupt JSON、udev rule 不一致、terminal 起動失敗を追加する。
16. `start.sh` の分岐を実行ベースで検証する
    - Linux 以外、`python3-venv` 不在、`pip install -e` 失敗をテストする。
17. README / docs を役割分担ベースに再編する
    - CLI がやること、手作業が必要なこと、hardware repo を見に行くタイミングを明示する。
